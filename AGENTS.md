# AGENTS.md

## 项目定位

本仓库用于设计并制作一辆低成本火星车原型（`PROJECT ARES / Operation MROS`）。项目已经从"纯方案设计"进入"结构件已建模、硬件已采购、准备装配与编程"阶段。仓库内容包括设计文档、手绘图纸、部件照片、3D 打印 STL、两块 ESP32 的固件、电脑端网页控制台和图像处理脚本。

后续新增内容默认应服务于"把这辆火星车装起来、跑起来、完成任务演示"的目标。

## 目标功能（所有文件提到的目标功能以此为主）

1. 六轮六驱，左右两侧差速转弯，总推力目标约 8 牛
2. 5m 距离能基本走直线
3. 能爬上 25° 的陡坡
4. 在 2m 高空坠落后仍能正常工作
5. 可自动驾驶
6. 可以使用机械臂采矿（列为第二阶段扩展，见 `docs/Ares_Control_Architecture.md` 第 12 节）
7. 制作材料不超过 150 美元
8. 工作寿命为 100 天以上（用休眠/唤醒计数模拟），并每天将当天数据转给 PC 端
9. 工作温度 -20°~50°（被动保温 + 温度监测 + 程序保护的模型化演示）
10. 任务目标：
    1. 拍摄 30 张 3D 图片（双目视差法：拍一张 → 直线前进 6cm → 再拍一张）
    2. 拍摄 10 张 360° 全景图片（SG90 舵机带动 ESP32-CAM 分步扫描，必要时配合车体自转补角度）
    3. 类似扫地机器人一样为预设区域制作一张地图（遥测航迹推算 + 超声波障碍点）
11. 机械 Rocker-Bogie 减震系统 + 2m 坠落生存设计（甲虫穹顶、硅胶减震球、软胎、弹性桅杆）

## 当前项目基线（V3，2026-07 定稿）

项目存在两代设计。**当前基线是 V3（手绘最新版）**，依据 `hardware/drawings/总体设计.jpg`、`hardware/drawings/线路图.jpg`、`docs/楼层布局.md`、`docs/2米高空坠落生存与自恢复方案.md` 和 `hardware/cad/` 下的 STL。用户已在 2026-07 拍板确认。除非用户明确要求，后续工作不得退回 V2.2 方案。

V3 基线要点：

- 双脑架构（不变）：
  `ESP32 NodeMCU` 主控（Commander）+ `ESP32-CAM` 视觉从板。
- 结构：
  3D 打印四层主舱（外廓 190 × 149 × **90**mm + 顶盖；旧 主舱.stl 的 75mm 装不下
  楼层叠高 5+25+4+35+4+主控板15=88，已按总体设计.jpg 的"9cm"修正）：B1 电池压舱层 → F1 动力配电层（L298N + LM2596）→ F2 神经中枢层（ESP32 + MPU6050 洞洞板）→ 天台（视觉桅杆 + 甲虫穹顶 + 横置钮子开关）。层间用尼龙铜柱 + 硅胶减震球，电池用 EVA + 魔术贴固定。
- 悬挂：
  `Rocker-Bogie` 摇臂-转向架，3D 打印摆臂（`hardware/cad/摆臂（需打印4份）.stl`），主铰链 A / 二级铰链 B 均为 Φ5 螺栓活动铰链，六轮方向固定，差速/坦克转向。
- 驱动：
  左右两侧各 3 个 TT 电机并联。**接线按单块 `L298N`**（OUT1/2 左组、OUT3/4 右组，见线路图.jpg）；每通道 3 电机并联堵转电流可能超过 L298N 单通道额定值，固件用软启动斜坡缓解，双 L298N 为备选升级（固件引脚不变），详见 `hardware/Ares_V3_GPIO_Map.md`。
- 传感：
  `MPU6050`（直线保持）+ **3 个独立 HC-SR04**（前/左前/右前三向感知，即 `AUTO-L2` 方案，不再采用 V2.2 的并联接法）+ 云台 `SG90` + 可选 `LDR`、`DS18B20`。
- 能源：
  `2 x 18650` 串联（7.4V）+ `2S BMS` + `LM2596` 降压 5V。**太阳能板已取消**（见 `docs/楼层布局.md`"取消太阳能板"），电池改用外部 8.4V 2S 充电器充电；折叠翼舵机随之取消。
- 打印件：
  以 `Print required/` 为唯一打印来源（13 件全部经 `hardware/cad/verify_fit.py`
  39 项装配校验）；改尺寸必须改 `hardware/cad/build_parts.py` 再重新生成，
  不要手改 STL。2026-07 上传的 9 个 `*_ARES_FIXED_V9.stl` 已确认存在孔径/
  对位/法线错误并删除。
- 引脚：
  以 `hardware/Ares_V3_GPIO_Map.md` 为全车唯一权威引脚表（电机 GPIO 16/17/18/19，三路超声波 13/14、27/26、33/32，I2C 21/22，舵机 4，UART2↔CAM 25/23）。V2.2 的 `Ares_Master_Wiring_Chart.md` 引脚分配已过时。
- 热控（不变）：
  被动保温盒 + 可选温度传感器 + 五态程序保护（NORMAL / COLD_START / COLD_SLEEP / HOT_LIMIT / THERMAL_PAUSE），不加无温控主动加热片。
- 抗坠落：
  电池 EVA 隔离、主控板硅胶减震球半悬浮、软胎（外径 65mm 空心/海绵胎）、TPU/弹簧弹性桅杆、X 状碳纤维甲虫穹顶被动翻正、钮子开关横置 + 防撞护翼。

## 文档冲突处理规则

两代方案冲突时的总原则：**V3 系（总体设计.jpg、线路图.jpg、楼层布局.md、坠落方案、STL）优先于 V2.2 系（完整指南、BOM、Wiring Chart）**。已裁决的具体冲突见 `docs/Ares_V3_决策记录.md`，包括：

- 太阳能板：已取消（V3），外部充电器充电。
- 超声波：3 个独立三向感知（AUTO-L2），不并联。
- L298N：接线按单块，双块为备选，固件不受影响。
- 充电保护：`2S BMS`，不用 `TP4056`（维持既有裁决）。
- 电机引脚：GPIO 16/17/18/19（V3），不是 32/33/25/26（V2.2）。
- I2C：代码按 SDA=21 / SCL=22；手绘图标注相反，若实物按手绘接线，改 `config.h` 两个常量即可。
- 机械臂：第二阶段扩展，不进当前基线。

如果发现新的不一致：优先采用该领域最专门的文档；优先采用与电压、电流、安全更一致的方案；在输出中写明哪些是文档确认、哪些是推断、哪些待用户拍板。

## 子系统职责划分

- 主控 ESP32（`firmware/commander/`）
  电机差速控制与软启动、MPU6050 直线保持、三向超声波避障、模式状态机（IDLE/AUTO/TELEOP-PC/CAPTURE/E-STOP/SLEEP）、热控状态机、休眠唤醒与任务计数、舵机云台、经 UART2 指挥相机板、经 WebSocket 与电脑端交互。是底盘运动的唯一仲裁者。
- ESP32-CAM（`firmware/cam/`）
  拍照、HTTP 上传照片到电脑、MJPEG 实时流、UART 应答（ACK/DONE/ERR/BUSY）。
- 电脑端（`dashboard/` + `vision/`）
  网页控制台负责遥测显示、手动接管、E-STOP、任务触发；Python 脚本负责收图、全景拼接、红蓝 3D 合成、轨迹/地图后处理、遥测记录；`mock_rover.py` 用于无硬件联调。

跨板通信接口（谁发命令、谁回确认）已在 `docs/Ares_Control_Architecture.md` 第 11 节定义，固件实现与其保持一致；修改协议时必须同步更新该文档、两侧固件和 dashboard。

## 工作原则

- 不要脱离现有文档随意换平台、换主控、换供电电压等级。
- 做设计判断时，优先保留文档中的系统意图，再谈优化。
- 每次新增方案、说明、代码前，先确认它属于哪个子系统。
- 如果提出改动，必须说明它影响哪些器件、接线、代码模块和测试项。
- 引脚改动必须先改 `hardware/Ares_V3_GPIO_Map.md`，再同步 `firmware/*/config.h`，两处不一致视为 bug。
- 如果信息不足，先补齐文档依据，再继续细化实现。

## 安全与工程底线

- 处理 `18650` 电池时严禁短路，必须经过 `2S BMS`；充电只用 2S 专用充电器（8.4V CC/CV），不得把太阳能板或其他电源直接接到电池上。
- 主控与传感器上电前，先确认 `LM2596` 输出稳定在 `5.0V` 左右。
- 所有模块共地；接地关系不清时，不要贸然出接线结论。
- `HC-SR04` 的 `Echo` 是 5V 电平，接 ESP32 前必须经 1kΩ/2kΩ 分压（3 路都要），ESP32 引脚不耐 5V。
- 调试电机时优先让车轮离地（架空），避免误动作跑飞。
- 涉及电池、充电的改动，默认按高风险任务处理；基线只采用被动保温和温度监测，不加无温控主动加热片。
- Rocker-Bogie 铰链必须能转动，不能用热熔胶粘死；电机固定优先用螺丝/扎带承重，热熔胶只做辅助防滑。
- 2m 跌落测试必须放在测试序列最后（T8），且先完成软胎、EVA、硅胶减震球、穹顶等全部防护再做；测试区铺缓冲垫，人员远离。
